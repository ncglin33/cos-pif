知識平台後台（KB）

建議的資料結構：kb_articles（專業文章）、 kb_terms（名詞/原料卡片）、 kb_refs（參考文獻）。
這頁採「匯入 → 預覽可編輯 → 批次寫入 Firestore」的工作流（沿用你現有匯入頁的操作邏輯）。

文章：kb_articles 名詞/原料：kb_terms 文獻：kb_refs

名詞 / 原料卡片（寫入 kb_terms）

推薦用於：濾劑/活性成分/檢測指標/法規名詞…（例如：SPF、UVA-PF、Niacinamide）。 CSV 匯入支援拖曳上傳或直接貼上。

將 Terms CSV 拖曳至此，或點擊選擇

尚未選擇檔案

或直接貼上 CSV 內容

必填欄位：nameZh 或 nameEn（至少一個）。
系統會以 type + nameEn/nameZh 產生穩定 docId；同一批匯入重複者，以最後一筆覆蓋。
type：term / ingredient / standard / regulation
topic/category：防曬、美白、抗老…
keywords/aliases：用逗號分隔

預覽（可點擊欄位直接修改）

專業文章（寫入 kb_articles）

文章建議用 JSON 匯入（可一次匯入多篇）。每篇文章含：標題/摘要/章節（markdown）/引用名詞/引用文獻/狀態。

將 Articles JSON 拖曳至此，或點擊選擇

尚未選擇檔案

或直接貼上 JSON（支援 JSON Array 或 JSON Lines）

必填欄位：slug、title。
docId 直接使用 slug（例如：whitening、antiaging）。
sections：[{id,h2,bodyMd}]
termRefs：["termId1","termId2"]
refIds：["refId1","refId2"]
status：draft/review/published

預覽（可點擊欄位直接修改）

參考文獻（寫入 kb_refs）

建議把 ISO、法規、期刊、教科書都統一進文獻庫，文章/名詞只要引用 refIds。

將 Refs CSV 拖曳至此，或點擊選擇

尚未選擇檔案

或直接貼上 CSV 內容

必填欄位：title（至少）。
docId 會依序使用：doi → url → title+year 產生穩定雜湊。
type：ISO/regulation/journal/book/guideline/web
tags：用逗號分隔
citationText：可留空（系統會自動簡易生成）

預覽（可點擊欄位直接修改）

建議的知識平台落地方式（最小可用 → 可擴充）

  1. 先做三集合：kb_terms / kb_articles / kb_refs（本頁已支援匯入）。
  2. 前台顯示：文章頁讀 kb_articles，名詞點擊彈窗讀 kb_terms。
  3. 搜尋：先用 keywords/aliases 做基本查詢；之後再加全文檢索或語意搜尋。
  4. 權限：前台只讀 published；後台需 editor/admin 才能寫入（用 Firestore Rules 管）。

重要：前台「可見性」建議完全用 status 控制（draft/review/published）。

Firestore Rules（簡化範例，請依你現有 roles 調整）

match /kb_terms/{id} {
  allow read: if resource.data.status == "published";
  allow write: if request.auth != null; // 建議改成 hasRole(editor/admin)
}
match /kb_articles/{id} {
  allow read: if resource.data.status == "published";
  allow write: if request.auth != null;
}
match /kb_refs/{id} {
  allow read: if true; // 文獻可以開放讀，或同樣只 published
  allow write: if request.auth != null;
}
